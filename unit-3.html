Unit 3 — Spot the Problem

The code runs. The output looks fine. Something is wrong.

What this unit is for

Everything you read in Units 1 and 2 was correct code. That ends now.

Each exercise below shows you a short analysis and the claim an analyst made from it. The code runs without errors and the output looks plausible — but in every case something is wrong: a mistaken assumption, a silent data problem, or a number that doesn’t mean what the analyst says it means. R will not warn you. The only defense is reading carefully and thinking about what the numbers should look like.

For each exercise:

  1. Describe what the code actually did (the Unit 2 skill).
  2. Identify what’s wrong.
  3. Explain why the output is misleading — ideally using what you know about the policy world, not just the code.

The dataset

We’re back to the ten-state health insurance data from Unit 1 — same columns, same values. It loads automatically; run the chunk to remind yourself what’s in it.

Exercise 1 — “The average state”

An analyst wants a single headline number for the briefing and writes:

“Across our ten states, the uninsured rate is 9.0%.”

The arithmetic is correct — the ten rates really do average to 0.09. Now think about what that number represents.

What’s the most important problem with calling this “the uninsured rate across our ten states”?

The population-weighted rate — total uninsured people divided by total population — comes out to about 9.7%, noticeably higher than 9.0%, because the biggest states (Texas, Florida, California) have above-average uninsurance. Whenever you see mean() applied to a column that is itself a rate, ask: should every row really count equally?

Try fixing it yourself — edit the chunk below so each state contributes in proportion to its population. (One approach: compute uninsured people with mutate(), then divide totals, like Unit 2’s Exercise 4.)

Exercise 2 — The unit trap

The team decides percentages read better than proportions, so an analyst rescales the column and then pulls out the “high-uninsurance” states for a map:

“Ten of our ten states have a high-uninsurance problem.”

Run the chunk. Why did all ten states survive the filter?

Write the note you’d send this analyst: what did the code actually do, why is the claim wrong, and what one-character-ish fix would repair it?

The code created a percent version of the uninsured rate (3–18 instead of 0.03–0.18) and then kept states where that percent exceeds 0.10. Since every percent value is far above 0.10, the filter kept all ten states — it did no filtering at all. The claim is wrong because the cutoff was meant on the old scale: “above 0.10” meant “above 10%,” but after multiplying by 100 the matching threshold is 10, not 0.10. The fix is filter(uninsured_pct > 10) (or filter on the original uninsured_rate > 0.10 before rescaling). The giveaway in the output is substantive: a “high uninsurance” list that includes Massachusetts, which famously has near-universal coverage, can’t be right.

A strong answer must (1) describe the rescaling and the filter, (2) pinpoint the unit mismatch — the threshold 0.10 is on the proportion scale while the column is now on the percent scale, so every state passes and the filter is vacuous, and (3) give the fix (compare against 10, or filter before rescaling). Bonus credit (not required for “strong”) for the domain sanity check that Massachusetts appearing on a high-uninsurance list is implausible. “partial” if the student senses the threshold is wrong but can’t articulate the scale mismatch or offers no fix. “needs_work” if they think the code errored, that mutate dropped rows, or that all ten states genuinely have high uninsurance.

Exercise 3 — The join that counted people twice

There’s a second table on this page, programs, listing state outreach programs for insurance enrollment — some states run several, some none:

An analyst joins it to the state data to size up the reach of outreach efforts:

“Outreach programs operate in states that are home to a combined 324 million people.”

That would be nearly the entire population of the United States — from ten states. Run the chunk and look at the rows count before reading on.

Describe what happened in the join, explain how the population total got to 324 million, and state what the correct combined population of these ten states is (roughly).

left_join() matches each state to its outreach programs, and when a state has several programs its row gets duplicated — once per program. California and New York each appear three times, Massachusetts and Texas twice, so the joined table has 16 rows instead of 10. Summing population over those 16 rows counts California’s 39 million three times, New York’s 20 million three times, and so on, inflating the total to about 324 million. The real combined population of the ten states is about 169 million. The lesson: after any join, check the row count before summing anything — duplicated rows turn sums into double-counts. (One more wrinkle: Pennsylvania and Nevada have no programs at all, yet they’re still in the total, so even “deduplicated” this number wouldn’t measure where programs operate.)

A strong answer must (1) explain that the join duplicates a state’s row once per matching program, growing 10 rows to 16, and (2) connect that to the inflated sum — populations of multi-program states are counted multiple times — and give roughly the right true total (anything near 169–170 million, or simply “about half the claimed figure,” is fine). The Pennsylvania/Nevada point (no programs but still counted, since left_join keeps unmatched rows) is bonus, not required. “partial” if the student sees the total is too big and blames the join but can’t explain the duplication mechanism. “needs_work” if they think sum() is the wrong function, that the population column is wrong, or that left_join dropped states.

The habit to take away

Notice what caught each error: not R skills, but expectations about the world. A 9% average that hides Texas, a “high-uninsurance” list containing Massachusetts, a ten-state population the size of the whole country. The code ran fine every time. Unit 4 hands you a full script and asks you to bring both skills at once — narrate it, and question it.
